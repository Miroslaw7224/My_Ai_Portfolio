Career Guide

Comprehensive automated testing infrastructure for the Career Guide app (FastAPI + React), designed and implemented from scratch. The project includes 745 tests, coverage across 13 API modules, CI/CD in GitHub Actions, and safety mechanisms preventing accidental use of production DB.

Status: Active development (Started: 2025 | In progress)

Role: Test automation engineer (primary contribution to test implementation, team collaboration)

About Career Guide

Career Guide is a career guidance platform that supports users from profile diagnosis through job applications and career growth.

For whom: people planning a career change, active job seekers, and users developing skills for specific roles.

Key product modules: user profiling (skills/experience/preferences), user-job matching algorithms, AI-assisted CV generator, psychometric module, application tracking and progress analytics, and premium feature payments.

Architecture: FastAPI backend (17 API routers), PostgreSQL, MinIO storage, React + Vite frontend, Supabase integration, and CI/CD in GitHub Actions.

Delivery context: team-developed project where my scope was designing and implementing the full automated testing infrastructure from scratch.

Problem -> Solution

Problem: The growing backend required repeatable and safe testing across many API modules, without risk of accidental impact on production.

Solution: Type-first test architecture (unit/integration/e2e/security), isolated Docker Compose environment, schema reset with safety guards, and CI/CD pipeline automatically triggered for PR and main.

Business impact: 0 production incidents, ~40% CI time reduction thanks to change detection, and easier onboarding for new testers through standardized workflow documentation.

Key Features

Type-first testing architecture
Test split into unit/integration/e2e/security layers for better scalability and suite readability.
Factories + fixtures ecosystem
11 custom factories and 43 conftest files with fixtures per API module.
Database safety guards
URL validation, production-host blocking, and schema reset with a hard guard for non-test DB.
Comprehensive E2E tests
E2E tests combining HTTP assertions with verification of real PostgreSQL changes.
Known regression handling
`xfail(strict=True)` markers to manage known regressions while preserving test suite integrity.
Automatic quality gates
GitHub Actions workflow blocking merges on failing tests and collecting coverage reports.

Key Technical Challenges

  • `sys.modules` contamination: moved from global mocks to per-test fixtures with module cleanup and fresh app imports.
  • PostgreSQL ENUM conflicts: replaced `drop_all()` with full schema reset (`DROP SCHEMA CASCADE` -> recreate).
  • Import path precision: standardized patching on full `backend.*` paths.
  • Order-dependent unstable tests: isolated known regressions with `xfail(strict=True)` markers.

How it works

LOCAL WORKFLOW:

1. Developer starts the test stack via Docker Compose with isolated `career_guide_test` DB and helper services.

2. The test runner executes the full pytest suite (unit, integration, e2e, security, smoke) with contextual fixtures and factories.

3. Safety guards validate the environment before `reset_schema` to eliminate risk of operations on non-test environments.

4. Coverage report is generated in `htmlcov/` for quick coverage-quality analysis.

CI/CD WORKFLOW:

1. PR/Push trigger starts GitHub Actions (`ci-pr.yml`).

2. Change detection optimizes test scope and pipeline runtime.

3. Docker stack with test DB and helper services starts automatically in CI job.

4. pytest suite runs as merge gate; pass/fail determines merge eligibility.

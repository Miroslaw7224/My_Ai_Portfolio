English Buddy

Full-featured app (PWA, responsive UI) combining a level-adapted conversational assistant, an audio flashcard system with review schedule, and skill assessment tools - with user accounts, optional error monitoring, and an admin QA panel.

// LIVE DEMO ↗ // GITHUB REPOSITORY ↗

Problem -> Solution

Problem: English learners often use fragmented tools: chat, vocabulary, reviews, and level assessment happen separately; there is no unified "tutor + exercises + feedback" experience synced with CEFR and long-term progress (SRS).

Solution: Next.js 15 app (App Router) with API routes, Supabase (auth + PostgreSQL with RLS), and OpenAI (chat, TTS, Realtime voice, agents/assessments). The user chooses or verifies level, learns through modules (chat, voice, flashcards, grammar, etc.), and progress/reviews are stored server-side according to profile configuration.

Key Technical Challenges

  • AI/CEFR consistency: Separate prompts and model behavior per level, while maintaining prompt hierarchy and assessment consistency.
  • User data security: PostgreSQL with RLS; sensitive operations run server-side only (e.g., Supabase service key).
  • Realtime audio: OpenAI Realtime integration with WebRTC and network fallback handling.
  • Learning content scale: Thousands of flashcards in JSON, QA pipeline (schema validation, smoke tests).
  • Demo mode and limits: Abuse mitigation (e.g., message/session caps) without degrading UX.
  • Production: Environment variables (OpenAI, Supabase, Sentry, cron secret), Vercel deployment, and public URL consistency (sitemap/robots vs legal docs).

How it works

1. Entry → Sign in via Supabase or start demo mode (without persistent progress storage).

2. CEFR level → Choose level or run placement test to verify level.

3. Modules → Chat, voice chat, flashcards, grammar, etc. from the main dashboard.

4. API → Frontend calls Next.js API Routes; server connects to OpenAI and/or Supabase.

5. State and persistence → Responses and learning events update state/cache (TanStack Query / Zustand), while persistent data (progress, SRS) is stored in DB.

6. Review cycle and admin → Review schedule and stats; administrator can use the QA panel (with permissions).

Career Guide

Kompleksowa infrastruktura testów automatycznych dla aplikacji Career Guide (FastAPI + React), zaprojektowana i wdrożona od podstaw. Projekt obejmuje 745 testów, pokrycie 13 modułów API, CI/CD w GitHub Actions oraz mechanizmy bezpieczeństwa chroniące przed użyciem produkcyjnej bazy danych.

Status: Aktywny rozwój (Rozpoczęty: 2025 | W trakcie)

Rola: Inżynier automatyzacji testów (główny wkład w implementację testów, praca zespołowa)

O aplikacji Career Guide

Career Guide to platforma doradztwa zawodowego wspierająca użytkownika od diagnozy profilu po proces aplikowania i rozwoju kariery.

Dla kogo: osoby planujące zmianę ścieżki zawodowej, kandydaci aktywnie szukający pracy oraz użytkownicy rozwijający kompetencje pod konkretne role.

Kluczowe moduły produktu: profilowanie użytkowników (skills/experience/preferences), algorytmy dopasowania user ↔ job offers, generator CV wspierany AI, moduł psychometryczny, śledzenie aplikacji i analityka postępów oraz obsługa płatności funkcji premium.

Architektura: backend FastAPI (17 routerów API), PostgreSQL, MinIO storage, frontend React + Vite, integracja Supabase i CI/CD w GitHub Actions.

Kontekst realizacji: projekt rozwijany zespołowo, a moim zakresem było zaprojektowanie i wdrożenie całej infrastruktury testów automatycznych od zera.

Problem → Rozwiązanie

Problem: Rosnący backend wymagał powtarzalnego i bezpiecznego testowania wielu modułów API, bez ryzyka przypadkowego wpływu testów na środowisko produkcyjne.

Rozwiązanie: Type-first architektura testów (unit/integration/e2e/security), izolowane środowisko Docker Compose, schema reset z guardami bezpieczeństwa oraz pipeline CI/CD uruchamiany automatycznie dla PR i main.

Wpływ biznesowy: 0 incydentów produkcyjnych, ~40% redukcja czasu CI dzięki wykrywaniu zmian oraz łatwiejszy onboarding nowych testerów dzięki standaryzacji i dokumentacji workflow.

Workflow rozkładu testów

Kluczowe Funkcje

Architektura testów type-first
Rozdział testów na warstwy unit/integration/e2e/security dla lepszej skalowalności i czytelności zestawu testów.
Ekosystem factories + fixtures
11 własnych factories i 43 pliki conftest z fixtures per moduł API.
Guardy bezpieczeństwa bazy
Walidacja URL, blokada hostów produkcyjnych i reset schematu z hard-guardem dla non-test DB.
Kompleksowe testy E2E
E2E łączące asercje HTTP z weryfikacją realnych zmian w PostgreSQL.
Obsługa znanych regresji
Markery `xfail(strict=True)` do zarządzania known regressions przy zachowaniu integralności test suite.
Automatyczne bramki jakości
Workflow GitHub Actions blokujący merge przy failing tests i zbierający raport pokrycia.

Kluczowe Wyzwania Techniczne

  • Zanieczyszczenie `sys.modules`: przejście z globalnych mocków na fixtures per-test z czyszczeniem modułów i fresh import app.
  • Konflikty ENUM w PostgreSQL: zastąpienie `drop_all()` przez reset całego schematu (`DROP SCHEMA CASCADE` → recreate).
  • Precyzja ścieżek importu: standaryzacja patchowania na pełnych ścieżkach `backend.*`.
  • Niestabilne testy zależne od kolejności: separacja known regressions markerami `xfail(strict=True)`.

Jak to działa?

WORKFLOW LOKALNY:

1. Deweloper uruchamia stack testowy przez Docker Compose z odseparowaną bazą `career_guide_test` i usługami pomocniczymi.

2. Test runner wykonuje pełny pytest suite (unit, integration, e2e, security, smoke) z kontekstowymi fixtures i factories.

3. Safety guards walidują środowisko przed `reset_schema`, żeby wykluczyć ryzyko operacji na środowisku nietestowym.

4. Coverage report generowany jest w `htmlcov/` do szybkiej analizy jakości pokrycia.

WORKFLOW CI/CD:

1. PR/Push trigger uruchamia GitHub Actions (`ci-pr.yml`).

2. Wykrywanie zmian optymalizuje zakres testów i czas wykonania pipeline.

3. Docker stack z testową bazą i usługami pomocniczymi startuje automatycznie w jobie CI.

4. pytest suite wykonuje się jako merge gate; wynik pass/fail decyduje o możliwości mergowania.
